ONOS源码学习笔记（一）

SDN控制器哪家强？学长强烈地安利了ONOS。

最近因为做一些SDN相关的东西，所以开始学习SDN。SDN，即软件定义网络，是一种新型的网络架构，核心思想是通过管控软件化，集中化，使网络变得更加开放、灵活、高效。具体来说，是将控制与转发流分隔开来，在控制面上为用户提供编程接口，在转发面上通过标准的协议接口（主要是Openflow协议）接收并执行转发策略。Openflow的关键技术又有Openfow交换机和Openflow控制器。

控制器是整个网络的大脑，通过增删交换机中的流表项动态地控制着转发策略。

而控制器从不同量级又分为很多种，有轻量级的pox，也有如onos这样的组件很多的。我们不禁要问：ONOS好处都有啥？谁说对了就给–打住。经过浅析ONOS的源码，我发现ONOS的组件非常齐全；并且有很完备的web界面；通过maven生成OSGi bundle，便于添加组件；跑在karaf容器中，便于动态activate和deactivate组件。总之，是很好用的一款控制器–如果看懂了ONOS的为数众多的组件的作用和工作原理的话。下面以ONOS众兄弟中控制转发的一个：onos-app-fwd为基础看看ONOS操作一个转发流程用到了哪些东西。

pom.xml

首先，在该组件的pom.xml中，组件的description为：

Reactive forwarding application using flow subsystem

这表明了组件的功能：主要是响应式的下发流表控制交换机转发。响应式表示控制器下发流表的动作是因为交换机收到不知道如何处理的包，向控制器上报而引起的。与之相对应有proactive forwarding，即控制器根据用户的configuration主动下发流表。

Felix注解

其次，开始看源码的时候，会发现里面有大量Apache felix的注解(annotation)，主要为这几个：

Annotation Usage Attributes
@Component 声明该class为一个组件（必需） immediate：是否立刻激活
@Reference 引用其他Service cardinality:强制还是选择性地引用；单引用还是多引用
@Property 定义一个属性，该属性通过ComponentContext.getProperties()得到 name:属性名字；value:属性值；label：标签
@Activate 激活组件时调用 无
@Deactivate 去活组件时调用 无
@Modified 改变组件时调用 无
@Service 暂且不管

这里需要对@Property特别关注一下，定义一个@Property时下面跟着一个跟name属性名字一样的field，这个field已经初始化为@Property注解中value的值，而代码中是这样获取该field的值的：

Boolean matchIcmpFieldsEnabled =
            Tools.isPropertyEnabled(properties, "matchIcmpFields");
if (matchIcmpFieldsEnabled == null) {
        log.info("Matching ICMP (v4 and v6) fields is not configured, " +
                 "using current value of {}", matchIcmpFields);
} else {
        matchIcmpFields = matchIcmpFieldsEnabled;
        log.info("Configured. Matching ICMP (v4 and v6) fields is {}",
                matchIcmpFields ? "enabled" : "disabled");
}

先通过属性的name获取属性，做一个空值判断，不为空就将值赋给该field，为什么需要这么麻烦，为什么不直接获取field的值呢？因为field已经初始化过了，而代码中也只对field进行过查询，没有重新赋值，什么情况下field可能变空值呢？暂时还不明白，此处打一个//TODO：

启动过程

下面看一下该组件的启动过程，也就是@ Activate注解的函数activate(ComponentContext context)。过程中出现的@Reference组件下一节写。

1.registerProperties(getClass())

这个函数由一个ComponentConfigService的实现类调用，用途是注册该组件中用到的properties。并把它们给一个叫ComponentConfigManager的类存起来，所有的组件的properties都需要给这个类存起来。

2.registerApplication(“org.onosproject.fwd”)

接下来由CoreService接口的该方法获取appId并返回。app名字的格式是reverse DNS格式。

3.addProcessor(processor, PacketProcessor.director(2))

这个方法由packetService调用，第一个参数是一个PacketProcessor类的实例，第二个参数是一个优先级，此处赋的优先级是一个director优先级，排在advisor之后，observer之前。
